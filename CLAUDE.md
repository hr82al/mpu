# CLAUDE.md

Этот файл — инструкции для Claude Code при работе в данном репозитории.

## Контекст проекта

`mpu` — multi-purpose CLI утилита. **Разрабатывается с нуля на Node.js.** Поведение, набор команд, форматы данных, конфиги и архитектура будут определены в процессе работы — **не копировать автоматически** из других проектов.

При работе с соседними кодовыми базами (`mpu-old/`, `/home/user/mr/mp/`) — использовать skill `reference-codebases`.

## Стек (согласован)

- **TypeScript + ESM**, Node >= 22. Вход — `src/bin/mpu.ts`, сборка в `dist/`.
- **commander** — CLI-фреймворк. Один bin `mpu` с subcommand; каждая subcommand — один файл в `src/commands/`, легко выносится в отдельный bin через `package.json#bin` при необходимости.
- **better-sqlite3** — синхронный SQLite (единый файл `$XDG_CONFIG_HOME/mpu/mpu.db` или `$MPU_DB`) для конфига и кэша.
- **Jest (ts-jest, ESM preset)** + `@jest/globals` (обязательный явный импорт в `.test.ts`). NODE_OPTIONS зашиты в `package.json` скрипты — запуск через `npm test [-- <jest args>]`.
- **ESLint flat config + Prettier** — конфиги в корне. `npm run lint`, `npm run typecheck`, `npm run format`.

## Структура (фиксированная)

```
src/
  bin/mpu.ts                    # shebang + buildProgram().parseAsync()
  program.ts                    # buildProgram() — собирает root + subs
  commands/
    <name>.ts                   # одна команда — один файл; экспортирует фабрику
    internal-complete.ts        # hidden runtime handler для автодополнения
  lib/
    paths.ts                    # defaultDbPath() — XDG-aware
    db.ts                       # openDb(), getDefaultDb(), миграции
    config.ts                   # Config + CONFIG_REGISTRY (registry с типом/default/описанием)
    cache.ts                    # Cache: get/set/delete/clear/wrap/wrapAsync
    completion.ts               # walker, провайдеры, emit bash/fish/zsh, installPath
    help.ts                     # describe(cmd, { summary, description?, examples? })
tests/                          # *.test.ts, выделены в отдельные файлы по модулям
```

Конвенции по областям лежат рядом с кодом и подгружаются автоматически:

- **Команды** (`describe()` для help, `setProvider()` для completion) → `src/commands/CLAUDE.md`
- **Библиотеки** (`Config`, `Cache`, `CONFIG_REGISTRY`) → `src/lib/CLAUDE.md`
- **Тесты** (`@jest/globals`, `:memory:`, конвенция «Проверяет: …») → `tests/CLAUDE.md`

При правках `lib/sheet-cache.ts` — использовать skill `sheet-cache`.
При выборе новой зависимости / решении «писать самому или взять библиотеку» — skill `dependencies-unix-way`.

## Что всё ещё НЕ определено

- Доменные команды mpu (webApp, client, ldb, rdb и т.д.) — ни одна ещё не реализована, **спрашивать перед созданием**.
- Env-переменные для внешних систем (Apps Script URL, sl-back API, PG) — пока не забраны из старого `.env`.
- REPL — нужен ли, и на чём (Node `repl`, WASM Janet, свой lisp, JS-eval) — открытый вопрос, не начинать без согласования.

## Общие принципы разработки

Эти правила применяются с самого начала, независимо от выбранного стека:

0a. **Любая команда чтения данных кэшируется.** Если команда обращается к внешнему API/сети за данными — её результат должен ложиться в кэш до возврата пользователю. Холодный вызов идёт в сеть, тёплый — в SQLite. Это обязательно для:
   - всех `get`-style команд (`sheet get`, `sheet ls`, `sheet info`, …);
   - любых будущих read-команд против внешних систем (sl-back API, PG-метаданные и т.п.).

   Реализация: либо общий `cache.wrapAsync(key, fn, { ttl })` в `lib/cache.ts` (плоское key→value), либо специализированный кэш с собственной семантикой (как `lib/sheet-cache.ts` для covering-fetch). Выбор зависит от того, можно ли склеивать/частично переиспользовать ответы — если да, нужен специализированный; если нет, достаточно общего.

   **Master switch — `cache.ttl=0`** отключает все кэши целиком (по соглашению из `lib/cache.ts`). Команды должны это уважать и при `ttl=0` ходить в сеть напрямую без сохранения.

   **Имя ключа** — `<область>:<идентификатор>[:<вариант>]`, например `sheet:info:<ssId>`, `sheet:keys:<ssId>:<sheet>`. Не вкладывать payload в ключ целиком — это ломает hits на семантически идентичных запросах.

0b. **Семантика > синтаксис: вывод проектируется по правилам, легко читаемым ИИ.**

   ИИ читает структурный вывод буквально. Если в ответе `formula: "5"` для ячейки без формулы (потому что Sheets API так возвращает в FORMULA-режиме), ИИ это считает «формула равна `5`», а не «формулы нет». Поэтому правила вывода:

   - **Ключ присутствует ⇔ его значение имеет смысл.** Не выставлять «синтаксический шум» (формула-эквивалент значения, пустая строка, дефолт). `formula` есть в ответе только если в ячейке **реально** формула (текст начинается с `=`). Иначе ключа нет.
   - **Унифицированные пустоты.** «Пустая/отсутствующая» данные — везде `null` (в JSON) и пустая строка (в TSV). Не смешивать `null`/`""`/`undefined` для одного семантического состояния.
   - **Не перегружать одно поле двумя смыслами.** Если поле может содержать значение или индикатор отсутствия — выделять отдельный флаг или менять схему.
   - **Каждое правило формулируется как предикат над ключом.** Хороший вывод даёт ИИ одно правило вида «есть `formula` → формула; нет → нет формулы». Плохой требует «`formula` есть всегда; формула, если значение начинается на `=`».
   - **Tabular-форматы (TSV) ровные по колонкам.** Колонки в TSV/CSV не выкидываются «по случаю», иначе индексы плывут. Колонка остаётся в header, ячейка пустая если не применима.

   Эти правила — для всех команд, не только `sheet`. При проектировании любого вывода спрашивать себя: «может ли ИИ ошибиться, читая это буквально?»

0. **Команды mpu — для нейросетей и людей одновременно.** Это базовый принцип, перекрывающий частные решения. На практике:
   - **Имена команд, флагов, ключей конфига — самодокументирующиеся.** `--spreadsheet`, а не `--ss-id`. `sheet get`, а не `bg`. Короткие алиасы (`-s`, `-n`) — поверх длинных, не вместо.
   - **`--help` команд — полный, со всеми примерами.** AI читает help перед вызовом; человек тоже. `describe()` хелпер обязателен.
   - **Ошибки — машинно-читаемые и человеко-читаемые одновременно.** Текст ошибки должен сам подсказывать, что попробовать дальше: какие env-переменные проверить, какие источники конфигурации просмотрены, какой формат ожидался. Никаких `unknown error` или `failed: 1` — всегда `<действие>: <причина> [<контекст: ssId, action, attempt, status>]; попробуй: <конкретная подсказка>`.
   - **Форматы ввода/вывода — гибкие и интуитивные.** Команды принимают позиционные аргументы, флаги, stdin (`--from -`), файл (`--from path`); не навязывают единственный путь. Вывод по умолчанию структурный (JSON) для AI/pipe, с переключателями `--raw`/`--tsv`/`--csv` для людей и shell-композиции. Если от формата зависит парсинг — указывать его явно в `--help` с примерами.
   - **Никаких "magic env" без документации.** Любая env-переменная, которую читает код, упоминается в `--help` команды или в общем help.

1. **Никаких лишних слоёв.** Не добавлять обёртки/абстракции "на будущее". Багфикс не тянет за собой рефакторинг. Три похожие строки лучше преждевременной абстракции. Перед тем как писать ~30+ строк инфраструктурного кода — проверить, нет ли подходящей библиотеки (см. skill `dependencies-unix-way`).

2. **Никаких "на всякий случай"**: не писать error-handling, fallbacks, валидации для невозможных сценариев. Доверять внутреннему коду и гарантиям фреймворков. Валидировать только на границах системы (пользовательский ввод, внешние API).
3. **Комментарии — только когда WHY не очевидно.** Не комментировать, что делает код (хорошие имена уже это показывают). Не писать "used by X" / "added for Y" — это устаревает.
4. **TDD** — любой новый функционал или багфикс начинается с падающего теста, воспроизводящего причину. Красный → минимум кода до зелёного → рефакторинг.
5. **DRY / Naming.** Перед написанием новой функции — искать существующую в этом же репо. Имя переменной отражает что хранится, а не намерение.
6. **Язык.** Тексты (README, комментарии к тестам, PR-описания) — по-русски, если пользователь не укажет иначе. Идентификаторы — по-английски.

7. **Строгая типизация.** `any`, `as unknown as T`, `@ts-ignore`, `@ts-expect-error` — запрещены без комментария с обоснованием на той же строке. Внешние границы (`JSON.parse`, `fetch`, sqlite rows, env, пользовательский ввод) типизировать через zod-схему или явный type-guard, не через каст. Внутри кода — доверять выводу типов, не «прибивать» типы избыточными аннотациями.

8. **Gate перед сдачей.** Прежде чем сказать «готово», прогнать локально:
   ```
   npm run typecheck && npm run lint && npm test
   ```
   Если хоть один шаг красный — задача не готова, чинить, не сдавать. Для CLI-команд дополнительно: запустить команду из `dist/` хотя бы один раз с реальными аргументами и показать вывод. typecheck+тесты ≠ работающая фича.

9. **Самопроверка перед сдачей.** Перед «готово» сам себе ответить:
   1. Что сломается, если этот код вызвать с пустым входом / `null` / очень большим входом?
   2. Что сломается, если внешний сервис ответит 500 / таймаутом / мусором?
   3. Если вызвать команду дважды подряд — поведение идемпотентно или ломается?
   4. Если кэш холодный vs тёплый — оба пути протестированы?

   Если на любой вопрос ответ «не знаю» — добавить тест или починить, прежде чем сдавать.

## Как начинать работу над задачей

1. Понять задачу от пользователя. Если она ссылается на старое поведение ("как в старом mpu") — прочитать соответствующий кусок `mpu-old/`, чтобы восстановить контекст, и **подтвердить с пользователем**, нужно ли такое же поведение в новой версии или другое.
2. Перед новой зависимостью, новым файлом в корне, новой схемой БД, новой таблицей, именами команд/флагов — **спросить пользователя до создания**.
3. Внутри согласованной структуры — действовать самостоятельно:
   - новая команда → `src/commands/<name>.ts` с фабрикой, `describe()` для help, `setProvider()` если нужно автодополнение значений, регистрация в `src/program.ts`
   - новый ключ конфига → добавить в `CONFIG_REGISTRY`
   - кэширование внешнего вызова → `cache.wrapAsync(key, fn, { ttl })`
   - тесты рядом с модулем в `tests/<module>.test.ts`
