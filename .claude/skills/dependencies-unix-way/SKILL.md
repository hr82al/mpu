# UNIX-way и зависимости

Не изобретать своё там, где есть отлаженная маленькая библиотека, делающая ровно одну вещь. Перед тем как писать ~30+ строк инфраструктурного кода, проверить — нет ли существующей библиотеки с правильной философией:

- **Do one thing well** — узкая ответственность, плоский API. Если у библиотеки 10 фич и 5 концепций — это framework, не лучший выбор.
- **Композируется** — принимает уже открытое соединение/файловый поток/массив, не диктует сборку.
- **Plain text > магия** — конфиги в `.sql` / `.json` / env, а не в DSL.
- **Зрелая** — годы в продакшене, малая площадь кода, малое дерево зависимостей. Не пакет с одним коммитом за 2024.
- **Sync если возможно** — текущий стек (`better-sqlite3`) синхронный; асинхронная библиотека форсит async на весь код.

## Текущие пробелы и рекомендации (использовать когда понадобится, не раньше)

| Пробел | Когда понадобится | Рекомендация |
|---|---|---|
| Миграции БД | первый `ALTER TABLE` или необходимость бэкфилла | `postgrator` — plain `.sql` файлы в директории, нумерованные, поддерживает better-sqlite3 через driver-инъекцию. Альтернатива — 30-строчный runner на `PRAGMA user_version`, если postgrator-async не вписывается. Текущий идемпотентный `CREATE TABLE IF NOT EXISTS` в `db.ts` пока достаточен. |
| .env-парсер | если понадобятся фичи свыше KEY=VALUE/quotes/comments (multiline, expansion) | `dotenv` — стандарт де-факто. Текущий `parseEnvFile` (~15 строк) пока достаточен, не подменять без нужды. |
| HTTP-ретраи общего назначения | если ретраи понадобятся вне webapp | сначала переиспользовать собственный `lib/retry.ts` — он чистый и инжектируемый. Сторонние (`p-retry`, `async-retry`) — только если потребуется их специфика. |
| CLI-аргументы | уже используется `commander` | не добавлять второй парсер. |
| SQLite | уже используется `better-sqlite3` | не добавлять ORM поверх — sql-строки прозрачнее. |

Когда добавляешь новую зависимость — фиксируй в этой таблице или в CLAUDE.md, чтобы будущий Claude не выбрал другую для той же задачи.
